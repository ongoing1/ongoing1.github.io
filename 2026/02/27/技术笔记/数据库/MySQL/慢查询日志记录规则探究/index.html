MySQL 慢查询日志记录规则探究

发表于2026-02-27|更新于2026-04-22|技术笔记数据库MySQL

背景

在排查数据库性能问题时，慢查询日志是常用工具。但有时看慢日志似乎并不能看到预期中的结果，于是有了以下思考：

  • SQL 语句被记入慢查询日志的判断依据究竟是什么？最直观的假设是：Query_time > long_query_time 时记录。但仔细想，当 SQL 因为等锁而变慢时，它还会被记录吗？Query_time和Lock_time 究竟是如何计算的？带着这个问题，这里设计了以下几组并发场景进行验证。
  • 实验环境：MySQL 5.7.29，long_query_time = 0.1

实验一：lock table 场景

t s1 s2 s3
1 lock table dba_test write;
2 select * from dba_test;
3 select sleep(1) from dba_test where id=1;
4 unlock table;
5 4 rows in set (8.16 sec) 1 row in set (3.18 sec)

说明

s1执行 lock table dba_test write; 会在mysql server层同时加两种锁：传统表锁和mdl写锁。s2执行select需要请求表的mdl读锁，读写互斥，因此被阻塞

结果

  • s2 未记录进慢查询日志，尽管从结果看它执行了8秒多
  • s3 记录进慢查询日志：Query_time: 3.181895 Lock_time: 2.181797

分析

  • MySQL 慢查询的记录规则是 Query_time - Lock_time > long_query_time。因此s2中sql没计入慢日志，因为这个sql实际执行时间不超过阈值
  • mdl 锁等待时间会被计入 Lock_time，解释了s3被计入慢日志后的Lock_time来源

实验二：alter table 锁场景

t s1 s2 s3
1 begin;
select * from dba_test;
2 alter table dba_test engine=innodb;
3 begin;
select sleep(1) from dba_test where id=1;
4 commit;
5 1 row in set (8.40 sec)
6 commit;
7 Query OK, 0 rows affected (26.24 sec)

结果

  • s3 记录进慢查询日志：Query_time: 8.405649 Lock_time: 7.405489
  • s2（ALTER）未记录进慢查询日志

分析

  • MDL 锁等待时间计入 Lock_time。但同时出现了两个新问题：
    • s2 耗时 26s 却没有任何记录——原因是 log_slow_admin_statements = OFF，ALTER/OPTIMIZE 等管理语句默认不参与慢查询记录。打开这个参数后，alter语句会计入慢查，Query_time: 26.248874 Lock_time: 22.400959
    • s3 在 s2（ALTER）之前执行完毕——s2 在 t2 就开始等待 MDL，排在 s3 前面，按照写锁高优先级机制，s1 提交后 s2 应先获得锁并执行 ALTER

实验三：rename table 场景

在 ALTER 场景的基础上，换用 RENAME TABLE 重复实验，出现了明显不同的结果：

t s1 s2 s3
1 begin;
select * from dba_test;
2 begin;
rename table dba_test to dba_test1;
3 begin;
select * from dba_test;
4 commit;
5 Query OK, 0 rows affected (42.75 sec)
6 ERROR 1146 (42S02): Table ‘op_test.dba_test’ doesn’t exist
7 commit commit

结果

  • s2 和 s3 均记录进慢查询日志，且 lock_time = 0.000000
  • s2（rename）先执行，s3 后执行（与实验二的顺序相反）

疑问

  • 为什么 RENAME 先执行，而 ALTER 后执行？
  • 为什么两条语句的 Lock_time 都是 0？

深入分析时间计算原理

关键变量

变量 设置时机 含义
start_utime 语句开始时 语句开始时间
utime_after_lock 获取锁完成后 锁获取完成时间
current_utime 语句结束时 语句结束时间

计算公式

// sql/log.cc:1382-1386
Query_time = current_utime - start_utime; // 总时间
Lock_time = utime_after_lock - start_utime; // 锁等待时间

时间线示意

关键源码

时间初始化 (sql/sql_class.h:3253-3255):
void set_time() {
start_utime = utime_after_lock = my_micro_time();
}

锁获取后更新时间 (sql/lock.cc:375):
// mysql_lock_tables() 函数末尾
thd->set_time_after_lock();

InnoDB 行锁等待时间累加 (sql/sql_class.cc:802-804):
void thd_storage_lock_wait(THD *thd, long long value) {
thd->utime_after_lock += value;
}

Lock_time 包含哪些锁等待时间

锁类型 是否计入 Lock_time 条件
MDL 锁（select和dml语句） 是 语句成功执行到 lock_tables()
MDL 锁（RENAME/DROP 等） 否 不调用 lock_tables()
表锁 (THR_LOCK) 是 句成功执行到 lock_tables()
InnoDB 行锁 是 通过 thd_storage_lock_wait() 累加

关键前提

Lock_time 只有在语句成功执行到 lock_tables() 时才会被正确统计。

以下情况 Lock_time = 0：

  • DDL 语句不调用 lock_tables()（如 RENAME、DROP）
  • 语句在获取锁阶段就失败了（如表不存在）

为什么 ALTER 和 RENAME 的执行顺序不同？

ALTER TABLE 的锁过程

ALTER TABLE INPLACE 分三个阶段，锁会升级和降级：

t1: s1 获得读锁
t2: s2(ALTER) 请求独占锁 → 等待 s1
t3: s3 请求读锁 → 排队等待
t4: s1 释放读锁
→ s2 获得独占锁（准备阶段）
→ s2 降级锁，允许读
→ s3 获得读锁，开始执行
t5: s3 执行完成并提交
→ s2 升级回独占锁（提交阶段）
t6: s2 完成

源码位置 sql/sql_table.cc:7594-7595：
// Upgrade to EXCLUSIVE before commit.
if (wait_while_table_is_used(thd, table, HA_EXTRA_PREPARE_FOR_RENAME))
goto rollback;

关键点： ALTER TABLE 在 MySQL 5.7 中默认使用 Online DDL（INPLACE 算法），该算法在准备阶段和提交阶段需要独占锁，但在执行阶段允许并发读写。

RENAME TABLE 的锁过程

RENAME TABLE 全程持有独占锁，不会降级：

t1: s1 获得读锁
t2: s2(RENAME) 请求独占锁 → 等待 s1
t3: s3 请求读锁 → 排队等待（独占锁优先级高）
t4: s1 释放读锁
→ s2 获得独占锁，执行 RENAME
t5: s2 完成，释放独占锁
→ s3 尝试获取读锁，但表已不存在，报错

关键点：RENAME 全程独占，s3 没有机会插队

为什么 RENAME 的 Lock_time = 0？

RENAME、DROP 等 DDL 语句不调用 lock_tables() 函数，只调用 lock_table_names() 获取 MDL 锁。

源码位置 sql/sql_rename.cc:146：
if (lock_table_names(thd, table_list, 0, thd->variables.lock_wait_timeout, 0))
goto err;

而 set_time_after_lock() 只在 lock_tables() 中被调用，所以：

  • utime_after_lock 保持初始值（= start_utime）
  • Lock_time = utime_after_lock - start_utime = 0

总结

回到最初的问题：SQL 被记入慢查询日志的判断依据是什么？

  1. 慢查询记录规则

    1. Query_time - Lock_time > long_query_time
    2. 等锁时间会被扣除。一条执行了 8 秒但全是在等锁的 SQL，不会被记为慢查询。

  2. Lock_time 包含哪些等待时间

    1. SELECT / INSERT / UPDATE / DELETE 语句：包含 MDL 锁、表锁、InnoDB 行锁的等待时间。
    2. RENAME / DROP 语句：Lock_time 恒为 0，因为不调用 lock_tables() 函数。
    3. ALTER 语句：包含等待时间，但默认不记录慢查询，需开启 log_slow_admin_statements = ON。

  3. ALTER 和 RENAME 的执行顺序为什么不同

    1. ALTER TABLE（Online DDL）：执行过程中会短暂释放独占锁允许读操作，后来的 SELECT 可以插队执行。
    2. RENAME TABLE：全程持有独占锁，后来的 SELECT 无法插队
