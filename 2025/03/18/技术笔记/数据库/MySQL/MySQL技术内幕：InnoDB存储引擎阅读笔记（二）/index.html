MySQL技术内幕：InnoDB存储引擎阅读笔记（二）

发表于2025-03-18|更新于2026-04-15|技术笔记数据库MySQL

第二章 InnoDB存储引擎

2.1 InnoDB存储引擎概述

InnoDB 是第一个完整支持 ACID 事务的 MySQL 存储引擎，由 Innobase Oy 公司开发，后被 Oracle 收购。

2.2 InnoDB存储引擎的版本

版本 MySQL 版本 新增特性
老版本 InnoDB MySQL 5.1 之前 内置，功能有限
InnoDB Plugin MySQL 5.1 可选安装，支持 Fast Index Creation
InnoDB 1.0.x MySQL 5.5 继承 Plugin 特性
InnoDB 1.1.x MySQL 5.6 支持全文索引、在线 DDL
InnoDB 1.2.x MySQL 5.7 更多性能优化

2.3 InnoDB体系架构（重点）

InnoDB 体系架构由后台线程和内存池两大部分组成。

后台线程

1. Master Thread

  • InnoDB 最核心的后台线程
  • 负责将缓冲池中的数据异步刷新到磁盘，保证数据一致性
  • 包括脏页刷新、合并插入缓冲、undo 页回收等

2. IO Thread

  • 使用 AIO（Async IO）处理 IO 请求
  • 共有 4 类线程：
线程类型 数量 作用
write thread 4（可配置） 写操作
read thread 4（可配置） 读操作
insert buffer thread 1 插入缓冲
log IO thread 1 日志写入

3. Purge Thread

  • 回收事务提交后不再需要的 undo log
  • InnoDB 1.1 版本开始从 Master Thread 独立出来

4. Page Cleaner Thread

  • InnoDB 1.2.x 版本引入
  • 将脏页刷新操作从 Master Thread 分离，减轻其负担

内存

1. 缓冲池 (Buffer Pool)

缓冲池是 InnoDB 中最大的一块内存区域，用于缓存磁盘数据。

工作原理：

  • 读取：首先检查缓冲池，命中则直接返回；未命中则从磁盘读取并放入缓冲池
  • 修改：先修改缓冲池中的页，再通过 Checkpoint 机制刷新到磁盘

缓冲池管理 - LRU 算法改进：

传统 LRU：最近使用的页放在列表头部
InnoDB 改进：midpoint insertion strategy

为什么这样设计？

  • 某些 SQL（如全表扫描）可能访问大量页但只用一次
  • 如果直接放到 LRU 头部，会将真正的热点数据挤出缓冲池
  • 新页先放 midpoint，只有在 old sublist 存活超过 innodb_old_blocks_time 后才能进入 new sublist

2. 重做日志缓冲 (Redo Log Buffer)

  • 默认大小 8MB
  • 三种情况下刷新到磁盘的重做日志文件：
    1. Master Thread 每秒刷新
    2. 每个事务提交时刷新
    3. 重做日志缓冲池剩余空间小于 1/2 时刷新

2.4 Checkpoint技术

为什么需要 Checkpoint？

Write Ahead Log (WAL) 策略：事务提交时先写重做日志，再修改页。

但存在问题：

  1. 缓冲池不够大，无法缓存所有数据
  2. 重做日志文件大小有限，循环使用
  3. 宕机恢复时间与重做日志大小成正比

Checkpoint 的作用：

  • 缩短数据库恢复时间（只需恢复 Checkpoint 之后的日志）
  • 缓冲池不够用时，将脏页刷新到磁盘
  • 重做日志不可用时，刷新脏页

两种 Checkpoint：

类型 触发时机 特点
Sharp Checkpoint 数据库关闭时 将所有脏页都刷新到磁盘
Fuzzy Checkpoint 运行时 只刷新部分脏页

Fuzzy Checkpoint 的四种情况：

  1. Master Thread Checkpoint：每秒/每10秒从脏页列表刷新一定比例脏页
  2. FLUSH_LRU_LIST Checkpoint：保证 LRU 列表有足够空闲页
  3. Async/Sync Flush Checkpoint：重做日志不可用时触发
  4. Dirty Page too much Checkpoint：脏页比例超过阈值（默认75%）时触发

2.6 InnoDB关键特性（重点）

1. 插入缓冲 (Insert Buffer / Change Buffer)

问题背景：

  • InnoDB 使用聚集索引，主键插入通常是顺序的
  • 但辅助索引（非聚集索引）的插入往往是随机的，导致大量随机 IO

解决方案：

  • 对于非唯一的辅助索引插入，先放入 Insert Buffer
  • 之后再以一定频率合并到真正的辅助索引中

使用条件：

  1. 索引是辅助索引（非聚集索引）
  2. 索引不是唯一的（因为唯一索引需要检查唯一性，必须读取）

Change Buffer（InnoDB 1.0.x 扩展）：

  • 不仅缓冲 INSERT，还缓冲 DELETE 和 UPDATE
  • 可通过 innodb_change_buffering 参数控制

2. 两次写 (Double Write)

问题背景：部分写失效（partial page write）

  • InnoDB 页大小为 16KB，操作系统页大小为 4KB
  • 如果写入过程中发生宕机，可能只写入了部分数据（如 4KB、8KB）
  • 此时页已经损坏，无法通过重做日志恢复（重做日志记录的是对页的物理操作）

解决方案：

恢复过程：

  • 如果页损坏，先从共享表空间的 doublewrite 区域恢复页
  • 再应用重做日志

3. 自适应哈希索引 (Adaptive Hash Index, AHI)

  • InnoDB 自动根据访问模式和频率创建哈希索引
  • 无需人工干预，完全自动化
  • 只能用于等值查询，不能用于范围查询

创建条件：

  • 以相同模式访问页 100 次以上
  • 页通过该模式访问了 N 次，N = 页中记录数 / 16

4. 异步IO (Async IO, AIO)

同步 IO：发出一个 IO 请求，等待完成后再发下一个
异步 IO：发出多个 IO 请求，无需等待，提高吞吐量

优势：

  • 可以进行 IO Merge：将多个相邻页的 IO 合并为一个
  • 例如：访问 (8, 6)、(8, 7)、(8, 8) 三个页，可合并为一次 IO

5. 刷新邻接页 (Flush Neighbor Page)

当刷新一个脏页时，检查该页所在区（extent）的所有页，如果是脏页则一起刷新。

优点：通过一次 IO 刷新多个脏页
缺点：

  • 不太脏的页可能又很快变脏
  • 对于 SSD 等随机 IO 性能高的存储，该特性收益不大

2.7 启动、关闭与恢复

innodb_fast_shutdown 参数：

值 含义
0 关闭时完成所有 purge 和 merge 操作（最慢）
1 默认值，不完成 purge 和 merge
2 只将日志写入日志文件，下次启动时恢复（最快）

innodb_force_recovery 参数：

  • 0：默认值，正常恢复
  • 1-6：不同级别的强制恢复模式，用于数据库无法正常启动时

本章核心要点总结

  1. 后台线程分工明确：Master Thread（核心）、IO Thread、Purge Thread、Page Cleaner Thread

  2. 缓冲池是性能关键：通过改进的 LRU 算法管理，避免热点数据被淘汰

  3. Checkpoint 保证持久性：将内存中的脏页刷新到磁盘，缩短恢复时间

  4. 五大关键特性：

    • Insert Buffer：优化辅助索引插入性能
    • Double Write：保证数据页完整性
    • AHI：自动建立哈希索引加速等值查询
    • AIO：异步 IO 提高吞吐量
    • Flush Neighbor Page：批量刷新脏页

疑难点标记

Q1: 为什么 Insert Buffer 只能用于非唯一辅助索引？

A: 唯一索引插入时需要检查唯一性约束，必须读取索引页，无法延迟。

Q2: Double Write 为什么要写两次？

A: 第一次顺序写入共享表空间（作为备份），第二次离散写入真正的数据文件。如果第二次写失败，可以从共享表空间恢复。

Q3: 自适应哈希索引和我们手动创建的索引有什么区别？

A: AHI 是 InnoDB 根据访问模式自动创建的内存结构，不需要用户干预，且只存在于内存中，重启后消失。

Q4: 缓冲池越大越好吗？

A: 在服务器内存允许的情况下，缓冲池越大，缓存命中率越高，性能越好。生产环境通常设置为物理内存的 50%-80%。
