第一章 MySQL体系结构和存储引擎

1.1 定义数据库和实例

数据库与实例的区别

概念 定义 组成
数据库 物理概念 操作系统上的文件集合（frm、ibd、MYD等文件）
实例 程序概念 后台线程 + 共享内存区
  • 在 MySQL 中，实例和数据库通常是一一对应的关系。一个实例就对应一份操作系统上的文件集合
  • MySQL 是单进程多线程架构

1.2 MySQL体系结构

MySQL 数据库区别于其他数据库的最重要特点：插件式的表存储引擎。

体系结构分层：

重要理解：

  • 存储引擎是基于表的，而不是基于数据库的
  • 同一个数据库中的不同表可以使用不同的存储引擎
  • 存储引擎负责数据的存储和提取，不解析 SQL（SQL 解析在上层完成）

1.3 MySQL存储引擎

InnoDB 存储引擎（重点）

InnoDB 是目前 MySQL 的默认存储引擎（MySQL 5.5.8 起）。

核心特性：

  • 支持事务：完整支持 ACID 特性
  • 行锁设计：支持行级锁，并发性能高
  • 支持外键：唯一支持外键约束的存储引擎
  • MVCC：多版本并发控制，实现高并发
  • 一致性非锁定读：默认读操作不加锁
  • next-key locking：避免幻读问题

存储方式：

  • 数据存放在逻辑表空间中（tablespace）
  • 可以配置为每张表一个独立的 ibd 文件（innodb_file_per_table）

MyISAM 存储引擎

MySQL 5.5.8 之前的默认存储引擎。

特点：

  • 不支持事务
  • 表锁设计：并发性能较差
  • 支持全文索引
  • 缓冲池只缓存索引文件，不缓存数据文件

文件组成：

  • .frm：表结构定义文件
  • .MYD：数据文件（MYData）
  • .MYI：索引文件（MYIndex）

其他存储引擎简述

存储引擎 特点 适用场景
NDB 数据存内存，Share Nothing 集群架构 MySQL Cluster 集群
Memory 数据存内存，重启丢失，哈希索引 临时表、快速查找
Archive 只支持 INSERT/SELECT，高压缩比 日志归档、审计数据
Federated 不存储数据，指向远程 MySQL 表 跨服务器访问

1.4 各存储引擎之间的比较

特性 InnoDB MyISAM Memory Archive
事务 ✅ ❌ ❌ ❌
锁粒度 行锁 表锁 表锁 行锁
MVCC ✅ ❌ ❌ ❌
外键 ✅ ❌ ❌ ❌
B+树索引 ✅ ✅ ✅ ❌
哈希索引 自适应 ❌ ✅ ❌
全文索引 ✅(5.6+) ✅ ❌ ❌
数据压缩 ✅ ✅ ❌ ✅

1.5 连接 MySQL

三种连接方式：

  1. TCP/IP（最常用）

    • 客户端和服务端可以在不同机器上
    • 通过 IP 地址和端口号连接

  2. 命名管道/共享内存（Windows 本地）

    • 配置 --enable-named-pipe 启用

  3. Unix 域套接字（Unix/Linux 本地）

    • 只能用于本地连接
    • 性能比 TCP/IP 略好

本章核心要点总结

  1. 数据库是文件，实例是程序：理解两者区别是理解 MySQL 架构的基础

  2. 插件式存储引擎：MySQL 最重要的特性，存储引擎基于表而非数据库

  3. InnoDB 是默认引擎：支持事务、行锁、外键、MVCC，适合 OLTP 场景

  4. 选择存储引擎的原则：

    • 需要事务 → InnoDB
    • 需要全文索引且 MySQL < 5.6 → MyISAM
    • 临时数据、快速查找 → Memory
    • 日志归档 → Archive

疑难点标记

Q1: 为什么说存储引擎是基于表的？

A: 创建表时可以指定 ENGINE=xxx，同一数据库的不同表可以用不同引擎。

Q2: InnoDB 的 MVCC 是什么？

A: 多版本并发控制，通过保存数据的历史版本来实现高并发读取，将在后续章节详细讲解。

Q3: 行锁和表锁有什么区别？

A: 行锁只锁定被访问的行，并发度高；表锁锁定整张表，实现简单但并发度低。
